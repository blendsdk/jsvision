# 00 — Ambiguity Register (Zero-Ambiguity Gate)

> **Plan**: layout-dsl-adoption/focus-traversal-primitive
> **Status**: ✅ GATE PASSED
> **Created**: 2026-07-19
> **CodeOps Skills Version**: 3.9.0

The plan is grounded in a two-part live investigation on branch `feat/dsl-adoptation` (session
context preserved): (1) a decode of the focus manager (`packages/ui/src/event/focus.ts`) + its spec
oracles, and (2) an **empirical characterization probe** that Tabbed through `FileDialog`/`ChDirDialog`
on the current code and recorded the focused-view sequence. The maintainer explicitly chose to *fix
the focus-traversal defect* rather than ship dialogs with degraded tab order ("yes, let's go" on the
"build focus primitive first" path). The items below record every decision with semantic weight; each
is Resolved with the maintainer's approved direction or a grounded single-viable default.

| # | Category | Ambiguity | Options considered | Resolution | Status |
|---|----------|-----------|--------------------|------------|--------|
| AR-1 | Behavior (core) | What is the new Tab traversal model? | (a) keep group-scoped wrap (status quo — traps); (b) **tree-order DFS that bubbles across group boundaries within the active scope**; (c) flatten dialogs (defeats the DSL) | **(b)** — the maintainer-approved fix. `advance()` walks tree order, exiting a group at its end into the parent's next focusable sibling; descent reuses `focusInto`. | ✅ Resolved |
| AR-2 | Behavior / correctness | What ceiling bounds the traversal (so Tab never leaks out of a modal or switches windows)? | (a) global mounted root (leaks out of a modal); (b) **`scopeRoot()` = `modal.topView()` while a modal is open, else the mounted root** | **(b)** — reuse the loop's existing dispatch scope (`event-loop.ts:492`). Traversal wraps at this ceiling; window switching stays a separate command (`desktop.focusNextWindow`). | ✅ Resolved |
| AR-3 | Behavior | On entering a container mid-traversal, go to its first/last focusable or its saved (last-focused) child? | (a) restore-or-first via `focusInto` for both directions; (b) always first/last (pure tree order, no memory); (c) **direction-aware descent + reset exited-group memory on Tab** | **(c)** — forward descent reuses `focusInto` (restore-or-first); reverse descent uses a new `descendLast` (restore-or-last), so Shift-Tab is the exact inverse of Tab. `advance` **resets the `current` of every group it climbs out of**, so continuous Tab is pure tree order (a wrap re-enters at the tree end, not the last-visited child — avoiding a relocated trap). Container **restore** is preserved for non-Tab entry (click / `focusView` / window switch / dialog open-close), which never runs the climb — so W-5 (`event.focus.impl.test.ts:29-52`) and the "focus survives switching windows / dialogs" contract hold. (a) rejected: it re-enters at the last-visited child, re-introducing a trap and skipping last children in reverse; (b) rejected: it breaks W-5's restore. | ✅ Resolved |
| AR-4 | Behavior | End-of-scope behavior. | (a) **wrap** to the first/last focusable in scope; (b) stop (dead-end) | **(a)** — wrap, matching today's per-group wrap and expected TUI behavior. | ✅ Resolved |
| AR-5 | Behavior | Tab / Shift-Tab with nothing focused. | (a) **Tab → first focusable in scope; Shift-Tab → last**; (b) no-op | **(a)** — matches the current entry behavior; preserves the `focusNext` "descend into scope" semantics. | ✅ Resolved |
| AR-6 | Scope (code) | Which symbols change? | (a) rewrite the focus manager; (b) **surgical: rewrite `advance()` + add one `descendLast` helper; add a `scope` parameter to `FocusManager.focusNext`/`focusPrev`; the loop passes `scopeRoot()`** | **(b)** — the change is confined to `focus.ts`: `advance()` gains the scope-ceilinged climb, resets the memory of the groups it exits (after capturing the prior-focused leaf so the blur still targets it), and a new `descendLast` (reverse mirror of `focusInto`) is added. `focusInto`/`focusView`/`getFocused`/`focusedLeafIn`/`isFocusable` are untouched. Public `EventLoop.focusNext()`/`focusPrev()` stay parameterless (they compute the scope internally), so callers/tests are unaffected. | ✅ Resolved |
| AR-7 | Testing (immutable-spec) | Are any existing focus spec oracles edited? | (a) edit some; (b) **none — additive only** | **(b)** — every existing focus spec/impl/hardening oracle passes **unedited** (verified: they test flat-wrap, single-nested descent, root-level restore, disabled-anchor recovery, modal quit-cascade — none asserts the deep-exit case being changed). New spec oracles are *added* for the DFS contract. No RD-02-style re-baseline is needed here. | ✅ Resolved |
| AR-8 | Kitchen-sink gate | Does this primitive need its own showcase story? | (a) new "focus traversal" story; (b) **covered — non-visual engine capability, exercised by the focus spec suite now and the file/chdir + ui/forms dialog stories downstream** | **(b)** — no new *visual component* is introduced; per the CLAUDE.md kitchen-sink scope, a non-visual capability gets a story only when meaningful to show standalone. The traversal is proven by ST-F1…F7 and demonstrated live once the flex dialogs land. | ✅ Resolved |
| AR-9 | Verification | Verify command + perf skip. | — | `yarn verify` (`TUI_SKIP_PERF=1` for the sanctioned non-gating editor-perf skip, as in Tier-0); `yarn bench` informational, no-regression check. | ✅ Resolved |
| AR-10 | Process / tracking | How is this tracked? | (a) fold into #120; (b) **new companion GitHub issue** (sibling to #117, "Primitive", out of the divergence set), a documented **prerequisite** of #115 + #120 | **(b)** — mirrors the #117 primitive-fix pattern; the roadmap gains a Primitive row and #115/#120 note the dependency. | ✅ Resolved |
| AR-11 | Behavior (guard) | Do widgets that use Tab internally regress? | — | No — but not for the reason a first pass assumed. An unbound `Tab` is intercepted in `route()` (`event/dispatch.ts:134`) and moves focus **before** any view's `onEvent` (the pre-process sweep is at `dispatch.ts:202`), so a view can **never** consume an unbound `Tab` in `onEvent`. A widget keeps `Tab` only by **keymap-binding** it — then it becomes a command at `dispatch.ts:124-129`, ahead of the focus step (the plan's Out-of-scope note states this correctly). The editor binds **no** `Tab` key (it uses Ctrl-Q/Ctrl-K prefix tables); "the editor lets Tab pass through" means it lets `Tab` reach `focusNext`, so the editor **is** subject to the tree-order change like every other widget. The real guard is the **Phase 4.1 full-verify sweep** plus enumerating any widget that keymap-binds `tab` (Phase 4.1). | ✅ Resolved |
| AR-12 | Performance | Reflow/alloc on the traversal path. | — | **No new per-frame allocation** — the traversal runs per Tab **keypress**, not per frame. Per press it does O(depth) small `children.filter(canReceiveFocus)` allocations (one per climbed level) plus O(depth) pointer writes for the exited-group memory reset, bounded by the scope subtree. Asserted by an impl test + `yarn bench` no-regression. | ✅ Resolved |
