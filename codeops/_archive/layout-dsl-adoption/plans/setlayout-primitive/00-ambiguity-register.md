| AR-2 | Which write sites migrate to `setLayout` here? | **The DSL internals + the `this.layout =` self-layout sites** — i.e. the DSL half of the issue's P1 plus all of its P2. The remaining composition writes belong to the open adoption issues that already own them; converting them here would duplicate that work, balloon the diff across 7 packages, and touch sites the epic is about to delete outright. ⚑ **Corrections, two.** (1) The DSL is **14** sites, not 13 — `dsl/flex.ts:217` (`spacer()`) was missed because its assignment wraps and a `.layout = ` grep cannot see it. | AR-3 | The subclasses that set layout via a **field initializer** (`override layout: LayoutProps = {…}`). | **Left declarative.** They cannot exhibit the footgun — a freshly-constructed object has no sibling props to drop — and converting them to constructor calls costs readability, forces constructors onto classes that have none, and shifts initialization order for no behavioural gain. Recorded precisely as the blocker P4 must solve. ⚑ **Corrections, two.** (1) There are **10**, not 11 (the 10 are enumerated in [02 §5.3](02-current-state.md); the count is wrong wherever the plan said 11, including the old AC-8). (2) A **second facet**: because a base constructor body runs *before* a subclass's field initializers, a `setLayout(…)` call inside a base constructor is silently erased by any subclass declaring `override layout = {…}`. There is no live instance today, but `setLayout` is about to be documented as *the preferred write path*, so the constraint belongs in its JSDoc — and P4 must solve it, not just the read-only field. | ✅ Resolved |
| AR-4 | Verify command at every phase gate. | `TUI_SKIP_PERF=1 yarn verify && yarn workspace @jsvision/examples test:e2e`. The second half is load-bearing: `yarn verify` runs only the `unit` project, and `ui` changes reach the demos through the built `dist`. ⚑ **Correction:** the `TUI_SKIP_PERF=1` prefix **does not reach the test processes**. `verify` fans out through turbo, `turbo.json` declares no `env`/`globalEnv`/`passThroughEnv`, and Turborepo 2.x defaults to Strict Environment Mode — so the variable is stripped and the perf assertions run regardless. This is at least the second plan in this feature to trip on it, so rather than work around it, **task 1.0 fixes the root cause**: add `"globalPassThroughEnv": ["TUI_SKIP_PERF", "CI"]` to `turbo.json`. The gate command is then the command it claims to be. | ✅ Resolved |
| AR-5 | `setLayout` auto-invalidates, but the DSL taggers do **not** today — so converting them changes runtime behaviour on already-mounted views. | **In scope.** At build time `host` is `null`, so the overwhelming majority of calls are unchanged no-ops. Pinned by ST-S5. ⚑ **Correction — the original justification was wrong and is withdrawn.** This entry claimed the change was a *latent bug fix* at one site, `filter-popup.ts:285`, "where the current silence is the defect". It is not: that write sits inside `this.bind(…, { relayout: true })`, and `View.bind` (`view.ts:243-245`) already calls `invalidateLayout()` after every apply — it reflows today. It is also not a tagger call at all, and it was simultaneously filed as one of the disjoint self-layout sites. The genuine tagger-on-a-mounted-view sites are `split/split-view.ts:186-190` (per splitter drag) and `datagrid/grid-panels.ts:563-564` (per `rebuildBody()`), and **both already reflow today — though by different mechanisms**: split-view through its own `bind(…, {relayout:true})` (`split-view.ts:170-171`), grid-panels **not** through its bind (`grid.ts:689` has no options object and is repaint-only) but through the `Group.add`/`Group.remove` inside `rebuildBody()` (`group.ts:93,115`). So **no tagger site observes this change today**, and the auto-invalidation stands as a **correctness default for future callers** — the alternative, a conditional invalidate, re-opens the exact "set and forget silently doesn't repaint" footgun FR-1 exists to close. ⚑ **Second correction (iteration 2): the change is not confined to taggers.** FR-5's self-layout conversions gain the same auto-invalidation, and **`window.ts:161` genuinely does add a reflow** — `commitPlacement()` does not invalidate today and runs on a mounted window at gesture start (`desktop.ts:213,222,235`). The delta is benign (it writes `bounds` back into `layout.rect`, so the reflow recomputes identical geometry) but it is real, and it is enumerated in [02 §3a](02-current-state.md) rather than absorbed into "no site observes it". ST-S5 remains valid and remains the one deliberate red **in Phase 2** — `fixed(v, 2)` on a mounted view genuinely does not invalidate today — but it **specifies new behaviour** rather than witnessing a repair. | ✅ Resolved |

The getter is part of P4's problem statement, not P1's. ⚑ And per AR-3's second facet, the getter alone is not even sufficient for P4 — it would not stop the 9 in-place `layout.rect = …` mutations. | ✅ Resolved |
