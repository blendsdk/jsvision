Zero-Ambiguity Gate register. Every design/behavior/scope decision below was presented with grounded
options and decided by the user. `(runtime)` tags mark items surfaced during authoring/execution.

| # | Item | Category | Options considered | Resolution (user decision) | Status |
|---|------|----------|--------------------|----------------------------|--------|
| AR-1 | Where the multi-click count lives | Architecture | (a) core decoder adds `clickCount` to `MouseEvent`; (b) UI loop stamps it on the `DispatchEvent` envelope; (c) per-widget shared `detectMultiClick` helper | **(b) UI loop stamps it** — computed once at the single envelope-enrichment seam (`route()` `ev2`, `dispatch.ts:183`) so **every** component reads `ev.clickCount` uniformly. (a) breaks core's pure/zero-dep/timestamp-free decoder + public event model + fixtures; (c) leaves per-widget boilerplate (defeats the framework-wide goal). **No `@jsvision/core` change.** | ✅ Resolved |
| AR-2 | Detection rule (what is the "same target") | Behavior | same **screen cell** (x,y) vs same **row index** | **Same screen cell** within the window — matches the editor's existing detector (`editor-mouse.ts:51`) and TV, whose `meDoubleClick` is position-based at the event layer. The loop stamps at dispatch time where only screen coords exist. For a fast double-click the list can't have scrolled, so same-cell ≈ same-row; a widget may still refine locally. | ✅ Resolved |
| AR-3 | Field shape on `DispatchEvent` | API | `clickCount?: number` (1/2/3…) vs `doubleClick?: boolean` | **`readonly clickCount?: number`** — expose the full consecutive count. Row widgets check `ev.clickCount === 2`; the editor's `smDouble`/`smTriple` map to 2/3 so it can later converge onto this one field. A boolean would cap the primitive at 2 and strand the editor's triple-click. | ✅ Resolved |
| AR-4 | Multi-click window + configurability | Behavior | 500 ms shared constant vs per-app `EventLoop` option | **500 ms shared constant** (`MULTI_CLICK_MS`, reused from the editor) — one framework-wide value, no new config knob. An **injectable loop clock** (`EventLoopOptions.now?: () => number`, default `Date.now`, mirroring `EditorOptions.now`) still lets headless tests drive exact timestamps. | ✅ Resolved |
| AR-5 | Tree single-click text emit | Behavior (TV fidelity) | drop it (TV-faithful) vs keep it | **Drop the single-click text emit** (`tree-rows.ts:261`) — TV's `TOutlineViewer` only activates on double-click (`toutline.cpp:465` `selected(foc)`); a single click just focuses / toggles the graph zone. Our port *added* the single-click activate. Restores fidelity and unifies the row family. Behavior change; documented. | ✅ Resolved |
| AR-6 | Which widgets consume `clickCount` in this plan | Scope | list/grid/tree only vs also editor/input | **Row family only** — `ListRows` (⇒ `ListView`/`ListBox`/`FileList`/`DirList`/`ComboBox`/`History` popups), `GridRows` (`DataGrid`), `TreeRows` (`Tree`). The **editor + input** local detectors stay as-is this plan and **converge onto `ev.clickCount` later** (out of scope, noted). | ✅ Resolved |
| AR-7 | Row activation trigger | Behavior | activate on `clickCount === 2` vs `>= 2` | **`=== 2`** — a second same-cell down activates; a 3rd (`clickCount 3`) does **not** re-fire. Single click still focuses+selects only (dropdowns depend on the click setting `selected`). | ✅ Resolved |
| AR-8 | `activateOnClick` opt-in flag | Scope | add a single-click-activates flag vs not | **Out of scope** — dropdowns already single-click-pick via their `selected`-watch; add the flag only if a real consumer needs it. | ✅ Resolved |
| AR-9 | File-dialog behavior | Behavior | new dialog code vs ride existing `openEntry` | **Ride the existing wiring** — `FileList.onSelect → openEntry` (`file-dialog.ts:242-256`) already enters a folder / resolves+closes a file; once `ListRows` activates on double-click, double-click a folder enters it and double-click a file chooses+closes, **no new dialog code**. | ✅ Resolved |
| AR-10 | `ComboBox`/`History` regression risk | Behavior | — | **No regression** — they watch the shared `selected` signal (`combo-box.ts:212-223`, `history.ts:120-131`); a single click already sets `selected` → picks + closes the popup. A double-click's 2nd down lands after the popup closed, so `clickCount` is purely additive (inert for them). | ✅ Resolved |
| AR-11 | Kitchen-sink coverage | Deliverable | new stories vs extend existing | **Extend** the existing `listview.story.ts` (id `containers/listview`) / `data-grid.story.ts` / `tree.story.ts` / `file-dialog.story.ts` blurbs to mention double-click; the smoke test keeps its mount-check role. Authoritative double-click behavior is tested at the loop + widget tiers (below), not the smoke. | ✅ Resolved |
| AR-12 | Verify command | Process | — | **`yarn verify` + `yarn lint`** (per CLAUDE.md); `TUI_SKIP_PERF=1` prefix permitted for the load-sensitive editor-perf budget. | ✅ Resolved |
| AR-13 (runtime) | Stamp mechanism + propagation | Design (runtime) | RouteContext seam vs loop pre-stamp on the envelope | **Loop pre-stamps `clickCount` onto the dispatch envelope for a mouse-`down` before `route()`**; `route()`'s `{ ...ev, …seams }` spread (`dispatch.ts:183`) and `hit-test.ts`'s `{ ...ev, local }` spreads (its `ev` param is bound to `ev2`) propagate it into `ev2` and the capture branch automatically. Computed on **`down` only** (undefined on move/drag/up/wheel/keys), matching TV's `meDoubleClick`. | ✅ Resolved |
| AR-14 (runtime) | Widget-unit-test seam | Testing (runtime) | inject a clock per widget vs pass `clickCount` on the envelope | **Bare widget unit tests set `ev.clickCount` directly** on the hand-built envelope (no clock needed) — the detector moved to the loop, so per-widget clocks are gone. **Integration tests through `createEventLoop`** use the loop's injected `now` and dispatch two real `down`s. Recorded because it deviates from #39's per-widget-clock assumption. | ✅ Resolved |
| AR-15 (preflight) | Graph-zone double-click on the Tree | Behavior (TV fidelity) | (a) reorder to activate-on-double in any zone (TV `meDoubleClick`-first); (b) keep zone-first, accept the deviation | **(b) Keep zone-first; a graph-zone double-click toggles twice (no activate), an accepted deviation from TV's `meDoubleClick`-first structure (`toutline.cpp:465`).** Our two-event model (cc1 then cc2) already toggled on the **first** down, so a single `meDoubleClick` can't be reconstructed; (a) would yield toggle-then-activate — still unlike TV. Text-zone double-click activates (the common, TV-faithful path). Surfaced by preflight **PF-002**: corrects the 03-02 GATE-2 claim from "matches ✅" to a documented deviation. | ✅ Resolved |
