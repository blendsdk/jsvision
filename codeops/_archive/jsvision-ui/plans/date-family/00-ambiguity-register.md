# Ambiguity Register — Date family (`Calendar` + `DatePicker`)

> **Feature**: jsvision-ui/RD-20 — Date family (`Calendar` + `DatePicker` + `CalendarDate`)
> **Gate status**: ✅ **GATE PASSED** — every item Resolved with an explicit decision; user confirmed.
> **CodeOps Skills Version**: 3.2.0

This register is the audit trail for the **plan-level** Zero-Ambiguity Gate. RD-20 itself passed
`add_requirement` + a `preflight` (see `../../requirements/00-preflight-report-RD-20.md`, ✅ PASSED,
8 findings resolved), so its scope decisions (AR-195…AR-209) are already user-confirmed and are
**not** re-litigated here. This register captures only the decisions the **plan** must make — the
ones RD-20 left as "confirmed at plan time" / "pinned at plan GATE-1", plus the two genuinely-new
design forks the RD deferred to the plan (the focus-cursor visual model, PA-1; the extension-role
byte set, PA-2).

Decisions carry `PA-n` ids (plan ambiguities); each cites the RD `AR-n` it descends from.

## How the gate was run

All 12 categories were swept against RD-20 + the real code (TV `calendar.cpp`/`calendar.h` +
`app.h`/`views.h` palette chain; ui `popup.ts`/`combo-box.ts`/`history.ts`/`picture.ts`;
core `theme.ts`/`palette.ts`). Most categories were already closed by the RD/preflight (scope,
grid geometry, selection/nav behavior, security, deferrals, demo). The open items clustered in:
**Naming & terminology / visual** (the focus-cursor model + the extension role bytes — PA-1/PA-2),
**Technical unknowns** (the anchored-popup generalization shape — PA-5; the file split — PA-6), and
**Ambiguities/source pins** (the GATE-1 decode + precedence + week-number algorithm — PA-3/PA-4/PA-10).
Zero items deferred. Two items (PA-1, PA-2) were **user-gated**; the rest are source- or
dominant-determined and recorded for traceability.
