# Ambiguity Register — Flexible Chrome Bars

> **Feature**: jsvision-ui / flexible-chrome-bars · **CodeOps Skills Version**: 3.3.2
> **Gate status**: ✅ GATE PASSED — all items resolved with explicit decisions.

The Zero-Ambiguity Gate register: every design/scope/behavior decision for the StatusLine +
MenuBar chrome-bar refactor, hunted across the 12 categories, each resolved to an explicit
decision. `[U]` = decided by the user in the planning interview; `[R]` = recommended by the planner
and confirmed (single viable path or low-stakes, user may correct). No item is deferred.

| #  | Category | Ambiguity | Decision | Status |
|----|----------|-----------|----------|--------|
| AR-1 | Scope | How deep do the menu-bar changes go? | **Layout only.** Right-align + spacer for menu **titles** and thread the flex-computed title x into the popup-anchoring controller. Embedded passive widgets (clock/indicator) **in the menu bar** are deferred to a follow-up. The menu controller's navigation model is untouched. `[U]` | ✅ Resolved |
| AR-2 | API / behavior | How does the status line accept both command items and embedded widgets? | **Everything becomes a View.** `statusItem()` returns a `StatusItemView`; `statusLine()` takes `View[]`. | AR-3 | API / behavior | How does a live status segment get dynamic text? | **Both mechanisms.** `statusItem` accepts `text: string \| (() => string)` (an accessor repaints the item on signal change), **and** any reactive view (`new Text(() => …)`) can be embedded. `[U]` | ✅ Resolved |
| AR-4 | Architecture | What is `StatusLine`'s base class + who owns interaction? | `StatusLine extends Group`; its children are the item/widget views laid out via `row`-direction flex (matches the `DataGrid`/`ListView` composite pattern). **StatusLine remains the single interaction owner** — it keeps `postProcess` mouse handling (press-capture, drag-retarget, emit-on-release), the accelerator sweep, and pointer capture; the child `StatusItemView`s are presentational (draw text + highlight from state pushed by StatusLine). `[R]` | ✅ Resolved |
| AR-5 | API | How is right-alignment expressed on each bar? | Status line: reuse the existing DSL `spacer()` (fr-weight fill) — zero new API. | AR-6 | Architecture | What is the "shared foundation"? | Both bars adopt the **RD-02 layout engine's 1-D flex solve** instead of a hand-rolled left-pack loop: the status line through a real `row()` of child views, the menu bar through a width-aware `layoutTitles` that calls the engine's `solveTrack`/apportion helper. One packing algorithm, two entry points. `[R]` | ✅ Resolved |
| AR-7 | Behavior / regression | Must default (no-spacer) geometry stay identical? | **Yes, byte-identical.** Status items pack from column 0, `gap:0`, ` text ` spans; menu titles pack from column 1 (`TITLE_MARGIN`) as ` name ` buttons.
