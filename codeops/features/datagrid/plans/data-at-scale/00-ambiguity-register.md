# Ambiguity Register — Data at Scale (RD-11)

> **Feature**: datagrid/RD-11 — Data at Scale (virtual scroll over a windowed source)
> **Gate status**: ✅ GATE PASSED
> **Created**: 2026-07-18
> **CodeOps Skills Version**: 3.8.0

Every semantically-distinct decision for this plan, hunted across the 12 gate categories and grounded
in the real code. Four forks were put to the user via `AskUserQuestion` (AR-1/4/5/6); the rest are
grounded defaults the user confirmed as a batch. The architecture (AR-2/3) was **red-teamed by an
independent challenger** before presentation — it converged the "length-correct lazy `display()`"
approach and caught that the datagrid body still depends on `display().length` through inherited
`GridRows` paths.

| AR-1 | Scope | How much of RD-11 this plan builds | Must-Have only / include pager now | **Must-Have only** — windowed virtual scroll + async loading + placeholders + per-frame coalescing + in-memory-large validation + story + showcase + security oracle. **Phase B (a later plan):** pager mode (`navigation:'paged'` + widget + page-size selector), infinite/lazy load-more, pluggable REST adapters, server aggregate. | AR-2 | Architecture | How the body reads a windowed source | (A) length-correct lazy `display()` skipping `materialize` / (B) sparse `(T\|undefined)[]` / (C) raw windowed `Proxy` with no guards | **(A)** On the windowed path only (`isWindowed = !!source.ensureRange`), `display()` returns a **length-correct lazy `Proxy`** (`.length === source.length()`, `[i] → source.rowAt(i)`), so every inherited `GridRows` read of `display().length` / `display()[i]` (`focusTo` `grid-rows.ts:317`, `super.updateTop` `:166`, `super.onEvent`/`activate` `:255,335`) keeps working with **zero `@jsvision/ui` change**. `materialize` is bypassed on this path; the eager path is byte-identical. The Proxy **fails loud** — any access other than `length` or an integer index (`.map`/`.find`/spread/iterate) **throws** a descriptive error rather than silently full-scanning or crashing, so any missed `isWindowed` gate surfaces as a located test failure ([preflight PF-003](00-preflight-report.md)). B smears `T\|undefined` across ~30 call sites and mutates the ui base config type; C page-faults every full-scan consumer and feeds placeholder objects to selection/footer as real rows. | ✅ | red-team |
| AR-3 | Reactivity | Repaint-on-resolve seam | reactive `revision?()` signal / `Promise` from `ensureRange` / plain counter | **Reactive `revision?()`** member on `GridDataSource`, read inside the `display` computed (mirrors the existing `version` signal at `grid.ts:338,416`) so a landed page re-derives `display()` to a fresh identity → the base display-bind repaints. A bare `Promise` bypasses the reactive graph (needs an imperative invalidate the body lacks) and a plain non-tracked counter silently never repaints in production. `ensureRange`'s `Promise` return is kept **only** for prefetch de-dup / test-settle. | ✅ | red-team |
| AR-4 | Windowed selection | Header select-all + tri-state over unloaded rows | disable for windowed / index-range selection model | **Disable select-all + tri-state for windowed** (they can't enumerate unloaded keys without page-faulting). **Ctrl/Shift range selection is also disabled for windowed** — the range path routes through `display().map(rowKey)` (`grid-selection.ts:159`, the same call gated off for select-all), which the loud Proxy cannot serve, and a placeholder row would call `rowKey(undefined)`. Only single-row **keyed** toggle (Ctrl-click / Space) on a **loaded** row survives, guarded in `GridSelection` (`toggleAtRow`/`rangeToRow`/`focusedKey`/Space-toggle: `rowAt(i) !== undefined` → no-op). Per-row keyed selection is unaffected (keys are stable across scroll). Index-range selection is a new semantics (out of scope). (Updated by [preflight PF-002](00-preflight-report.md).) | ✅ | user |
| AR-5 | Windowed mutation | insert / delete / duplicate on a windowed source | append+delete via seam, no positional / read+edit only / full server mutation | **Append + delete via seam; no positional.** `insertRow` appends when `source.insert` exists; `deleteRows` via `source.remove(keys)`. `duplicateRow` + positional insert-at (which client-scan the source, `row-mutations.ts:89,105`) are **no-ops + `devWarn`** for windowed. | ✅ | user |
| AR-6 | Placeholder UX | How an unloaded row renders | muted `…` per cell / dim skeleton bar / blank muted row | **Muted `…` per cell** in a muted role — reads as 'loading', no layout shift. | ✅ | user |
| AR-7 | Push-down | Sort/filter/distinct on a windowed source | require push-down / allow client fallback | **Require** `setSort` + `setFilter` (+ `distinct` for value-lists) whenever `ensureRange` is present. Missing → **construction throws** (hard-fail, not a `devWarn`): client sort/filter over partial data is meaningless, and the `applySort`/`applyFilter` re-anchor scans (`grid.ts:1234-1240`/`1254-1261`) are gated by push-down *presence*, not `isWindowed` — a bare `devWarn` would let a mis-built windowed source reach `.findIndex` on the loud Proxy and throw at an unhelpful site. The client-scan path (`materialize` + `filterRows`/`sortRowsMulti`, `grid.ts:417-419`) **never runs** for a windowed source. (Hardened by [preflight PF-003](00-preflight-report.md).) | ✅ | confirmed |
| AR-8 | Auto-width | Column widths on a windowed source | force fixed/`fr` / measure page-0 once | **Force fixed/`fr`.** `measureAutoWidths` (`grid.ts:422`) is not invoked over a windowed source; an `auto` column falls back to a fixed default + `devWarn`; `autoFitColumn` (`grid.ts:1019`) no-ops for windowed. Prevents reflow-on-scroll (the spike's proven constraint). | ✅ | confirmed |
| AR-9 | Footer | Aggregates on a windowed source | loaded-window fold + "(loaded)" / **defer to Phase B** / server aggregate | **Windowed footer aggregates deferred to Phase B.** For a windowed source (`ensureRange` present) an aggregate cell renders **blank + a one-time `devWarn`** — there is no loaded-range seam to fold cheaply (the plan adds only `revision?()`), folding via `rowAt(0…length())` would page-fault/fetch-storm, and a no-eviction "loaded" fold drifts upward on scroll (misleading). The `complete()`/"(loaded)" honesty seam (`grid-footer.ts:107`) still serves the existing partially-loaded **eager** path unchanged. A server-side grand-total aggregate is Phase B. (Revised from a loaded-window fold by [preflight PF-001](00-preflight-report.md).) | ✅ | confirmed → revised (PF-001) |
| AR-10 | Counts | `filteredCount` / `totalCount` on windowed | read `source.length()` / add a filtered-vs-grand split | **`filteredCount()` reads `source.length()`** on the windowed path (not the materialized display length). | AR-11 | Prefetch | Buffer size + coalescing | one viewport each side, configurable, ≤1/frame | **One viewport-height** of rows above and below the visible window, configurable via a new `prefetch?: number` option (rows); **unset ⇒ the buffer is the current `visibleRows`, resolved per-draw** — the option carries no static default because the viewport height is dynamic ([preflight PF-008](00-preflight-report.md)). ## Notes

- **Same-session authoring:** this plan was authored in the session that will (later) preflight it.
  A fresh-session preflight is recommended for review independence. The load-bearing architecture
  (AR-2/3) was already run past an independent challenger during authoring.
- **RD ARs inherited:** RD-11's own Scope Decisions (RD AR #4 scroll-default, #14 windowed-strategy,
  #28 pager-backend-deferred, #1 adapters-generic) are honoured by AR-1/2/13 here.
- **Preflight resolutions (2026-07-18, iteration 1 — [00-preflight-report.md](00-preflight-report.md)):**
  a fresh-session preflight resolved 11 findings (5 major, 4 minor, 2 observation), all accepted. The
  load-bearing corrections: **AR-2** — the windowed Proxy now **fails loud** (whole-array ops throw),
  because empirically an ungated `.map`/`.find`/spread does **not** yield an empty array — it throws on
  the first unloaded row or fetch-storms the dataset (PF-003). **AR-9** — the windowed footer fold is
  **deferred to Phase B** (blank + `devWarn`); it was infeasible without a loaded-range seam (PF-001).
  **AR-4** — Ctrl/Shift **range** selection is disabled for windowed, not kept (PF-002). **AR-7** — the
  push-down requirement **hard-fails construction** instead of warning (PF-003). Plus fixture-ordering
  (a new task 1.0, PF-004), the kitchen-sink registry retarget (PF-005), and minor spec-oracle rewrites
  (ST-12/13/14/15). ST oracles were revised **here, pre-execution** — before any immutable test is
  authored — which is precisely preflight's job.
- **Runtime decisions (exec_plan, 2026-07-18):** items tagged `(runtime)` in the Category column were
  resolved during execution per the Zero-Ambiguity mandate. **AR-18** — the eager `windowed-source.ts`
  fixture's no-op `ensureRange` was removed so it isn't misclassified as windowed (a zero-regression
  precondition the plan hadn't covered). Decided with the user before Phase 1.
