## Overview

A professional, DX/UX-first documentation and showcase website for JSVision, built with VitePress
and deployed to GitHub Pages. It is the project's **sales pitch** and its **teaching surface**: an
inviting landing page, per-component tutorials, generated API reference, best-practices and FAQ
content, and four polished sample applications.

Its defining feature is that **every code example runs live, in the browser, in a real terminal**.
This is possible because the JSVision engine is pure "bytes in → bytes out" (`serialize()` produces
ANSI, `decode()` consumes ANSI) — precisely xterm.js's contract. The `packages/examples/web-xterm/`
spike already proved the entire stack runs client-side by swapping only the OS boundary. This
feature-set extracts that proof into a first-class, tested `@jsvision/web` package and builds the
whole site on top of it, with **no web backend** of any kind.

The site must never silently drift from the code: every docs example is a real, compiled,
headlessly smoke-tested module embedded by snippet (never copy-pasted), and a hard-failing CI gate
(`check:docs-site`) blocks any merge where a public component lacks a docs example or an embedded
example fails to compile or render.

## Domain Glossary

| Term | Definition |
|------|-----------|
| **Live example** | A JSVision app module that runs in the reader's browser inside an xterm.js terminal, driven by `@jsvision/web` — the same engine as a native terminal, no backend. |
| **Play button** | The docs UI affordance that opens a modal dialog hosting a live example in an xterm.js terminal, instantiated lazily (only when opened) and disposed on close. |
| **DemoShell** | A reusable JSVision application shell wrapping every demo and sample app: a menu bar with an **About** dialog and a **Theme switcher** (all 13 presets, default Turbo Vision), plus a status line. Single-component demos use it too — nothing is shown "naked". |
| **`@jsvision/web`** | The new browser-runtime package: the xterm.js host (out=`serialize`→`term.write`, in=`term.onData`→`decode`), a browser in-memory **virtual FileSystem**, key-chord reclaim, and a clipboard bridge. Private until first release. |
| **Virtual FileSystem** | A browser, in-memory implementation of `@jsvision/files`' injectable `FileSystem` interface, seeded with a sample project tree, so `FileDialog`/`ChDirDialog`/`FileList`/`DirList`/tvedit run unchanged in the browser. |
| **Snippet embed** | VitePress's file-import of a source region into a fenced code block, so docs show real source that is compiled and tested, not hand-copied text. |
| **`check:docs-site`** | The hard-failing CI gate enforcing docs↔code parity: every public component has a docs example, and every embedded example compiles and passes its headless smoke test. |
| **Public surface** | The exports of each package's `index.ts` barrel — the components/primitives that get a doc page and appear in the API reference (~40 across `@jsvision/ui`, `@jsvision/files`, and re-exported core primitives). |
| **Key-chord reclaim** | Intercepting browser-hijacked chords (F1, F5, Ctrl+W, Ctrl+N, Tab, Alt+letter, …) inside a focused live terminal via `attachCustomKeyEventHandler` + `preventDefault`, so they reach the TUI instead of the browser. |

## Key Architecture Decisions

| Decision | Choice | Rationale | AR |
|----------|--------|-----------|----|
| Live-demo runtime | Client-side only (xterm.js), no backend | Proven by the web-xterm spike; static hosting; scales infinitely; strongest pitch | AR-1 |
| File dialogs in browser | Browser virtual `FileSystem` + upload/download bridge | `@jsvision/files` already injects `FileSystem`; runs unchanged | AR-2 |
| Docs examples | Separate from kitchen-sink, but real + smoke-tested + snippet-embedded | User choice + anti-drift | AR-3 |
| Browser host | Extract to `@jsvision/web` (first-class, tested) | Reusable + testable; a real "JSVision in the browser" surface | AR-6 |
| API reference | TypeDoc → markdown → VitePress | One theme/nav/search; CI-regenerated from public surface | AR-7 |
| Deploy | GitHub Pages project subpath + PR previews | No vendor; `base:'/jsvision/'`; test each slice live | AR-4, AR-8 |
| Site placement | New `packages/docs-site`, absorb existing `docs/` techdocs | One site; workspace can bundle live demos | AR-23 |
| Coverage | Full public barrel surface (~40) | Meets "every component"; gate-enforceable | AR-27 |
| Security posture | Full: CSP, sandboxed future REPL, client-only uploads, `sanitize()` | "Security is a selling point" | AR-26 |
