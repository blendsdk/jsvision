# Site Integration & Cross-Linking: API Reference

> **Document**: 03-02-integration-cross-linking.md
> **Parent**: [Index](00-index.md)

## Overview

This document owns how the generated tree joins the VitePress site — the imported sidebar, the
hand-written preface, and the **bidirectional** component↔reference cross-links driven by one
symbol↔page map (AR-4/13). The generated markdown itself is owned by
[03-01](03-01-generation-pipeline.md); this is everything a reader navigates.

## Architecture

### Sidebar wiring

`typedoc-vitepress-theme` writes `api/typedoc-sidebar.json` — a VitePress sidebar array. The config
loads it for the `/api/` route instead of the current one-item static entry. Because the JSON is
gitignored and produced only by `docs:api` — and `docs:dev` does **not** chain `docs:api` (only the root
`docs:build` does, AR-14) — the load is a **defensive fs-read**, not a static `import … with { type:
'json' }` (AR-19). A static import hard-fails on a fresh checkout (the file is absent) and can't be
guarded; the fs-read falls back to an empty array so `yarn docs:dev` and a direct `vp:build` both start:

```ts
// .vitepress/config.ts (near the static sidebar object / the /api/ entry)
import { existsSync, readFileSync } from 'node:fs';
import { fileURLToPath } from 'node:url';

// Generated by `docs:api`; gitignored. Absent on a fresh checkout / before the first generation,
// so fall back to an empty array rather than hard-failing config load. The anti-drift gate
// guarantees the shipped build (which runs `docs:api` first) always carries the real tree.
const sidebarPath = fileURLToPath(new URL('../api/typedoc-sidebar.json', import.meta.url));
const typedocSidebar = existsSync(sidebarPath)
  ? (JSON.parse(readFileSync(sidebarPath, 'utf8')) as unknown[])
  : [];
// …
sidebar: {
  // …existing sections…
  '/api/': [
    { text: 'Overview', link: '/api/' },              // the hand-written preface (AR-13)
    ...typedocSidebar,                                  // generated: per package, grouped by kind
  ],
}
```

Under the root `docs:build` the generated JSON exists before the VitePress build (it chains `docs:api`
first — AR-14), so CI always renders the full tree; the fallback only covers the not-yet-generated dev
case. The top-nav `{ text: 'API', link: '/api/' }` is unchanged.

### The API preface — `api/index.md` (AR-13)

The committed placeholder is rewritten as the "how to read this" preface: what the reference is, that
it is generated from the public `index.ts` surface, how it is grouped (package → kind), and a
**pre-release note** naming `@jsvision/ui` / `files` / `web` as private-until-release (AR-6). It links
into each package's generated root. No generated content is hand-copied here.

### The symbol↔page map — `packages/docs-site/src/api/api-map.mjs`

One module is the single source of truth for both link directions. It is **plain ESM** (`.mjs`, AR-18)
so the plain-`node` `gen-api.mjs` can `import` it directly — a `.ts` module would need a loader `node`
does not have. The `ApiLink` shape is a JSDoc `@typedef` (documented + editor-checkable) rather than a
TS `interface`:

```js
// packages/docs-site/src/api/api-map.mjs
/**
 * @typedef {object} ApiLink
 * @property {string} symbol              The exported symbol name, e.g. 'Button' or 'createApplication'.
 * @property {'core'|'ui'|'files'|'web'} pkg  Unscoped package, e.g. 'ui'.
 * @property {string} apiPath             Site-absolute path to the generated symbol page, e.g. '/api/ui/classes/Button'.
 * @property {string} componentPage       Site-absolute path to the hand-written component page, when one exists.
 */

/** @type {ApiLink[]} */
export const API_MAP = [ /* seeded from the components that already have pages */ ];
```

Seeded on the components that already have RD-05 pages (Button, Input, DataGrid, Calendar, …), it is
extended by adding rows — not by touching the mechanism. `apiPath` values are validated against the
real generated tree by the gate (ST-11), so a plugin path-scheme change fails loudly rather than
silently 404-ing.

### Forward links: component page → reference (AC-5 forward)

Each mapped component page gains an "API reference →" link in its existing `## Related` section,
targeting `apiPath`. These are hand-authored into the (committed) component pages but kept honest by
ST-11 (every `API_MAP.apiPath` resolves to a generated page) and by VitePress's own dead-link check.

### Back-links: reference → component (AC-5 back, Must-Have bidirectional)

A pure injector runs as the second stage of `gen-api.mjs` (after TypeDoc writes the tree):

```js
// packages/docs-site/src/api/inject-back-links.mjs  (pure; unit-spec'd — 03-03)
/**
 * Given a generated page's markdown and the ApiLink for its symbol, return the markdown with a
 * '> **Documented in:** [<page>](<componentPage>)' note inserted after the page frontmatter.
 * Idempotent: a page that already carries the note is returned unchanged.
 * @param {string} markdown
 * @param {import('./api-map.mjs').ApiLink} link
 * @returns {string}
 */
export function injectBackLink(markdown, link) { /* … */ }
```

The wrapper walks `API_MAP`, reads each `apiPath`'s generated file, applies `injectBackLink`, and
writes it back. Because generation is gitignored + rerun every build, the injected back-links are
always regenerated — never committed, never stale.

## Implementation Details

### New / changed files

| File | Kind | Purpose |
| ---- | ---- | ------- |
| `packages/docs-site/src/api/api-map.mjs` | new | The `API_MAP` source of truth (forward + back); plain ESM (AR-18/20) |
| `packages/docs-site/src/api/inject-back-links.mjs` | new | Pure `injectBackLink` (idempotent); plain ESM (AR-18) |

| `packages/docs-site/components/**/*.md` (mapped) | edit | Forward "API reference →" link in `## Related` |

## Integration Points

- **Consumes** `api/typedoc-sidebar.json` + the generated trees from
  [03-01](03-01-generation-pipeline.md).
- **`injectBackLink`** + the `API_MAP` extractor are unit-spec'd by
  [03-03](03-03-gating-anti-drift.md); path-resolution is gated by ST-11.

## Error Handling

| Error Case | Handling Strategy | AR Ref |
| ---------- | ----------------- | ------ |
| `API_MAP.apiPath` points at a non-existent generated page | ST-11 fails the build gate (before a dead link can ship) | AR-4 |
| Back-link injected twice (rerun) | `injectBackLink` is idempotent — a page already carrying the note is returned unchanged (ST-7) | AR-4 |
| A component page's forward link 404s | VitePress dead-link check fails `docs:build`; ST-11 catches it earlier | AR-4 |
| Generated sidebar JSON missing at config load (`docs:dev` on a fresh checkout, or a direct `vp:build`) | The defensive fs-read falls back to `[]` so config load never hard-fails; the `/api/` route shows only the Overview until `docs:api` has run. The shipped build always runs `docs:api` first (root `docs:build`), so CI/production carry the full tree | AR-19 |

## Testing Requirements

- `injectBackLink` pure-function behavior + idempotency (ST-7).
- `API_MAP` completeness/shape (every row's `componentPage` exists; no dup symbol) (ST-12).
- Every `apiPath` resolves in the generated tree (ST-11).
- The generated pages render in-site and appear in local search (ST-8, ST-9).
