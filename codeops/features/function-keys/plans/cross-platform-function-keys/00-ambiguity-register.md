### Resolution Notes

**AR-1:** The planning target is the new nested-layout feature `function-keys`; context includes the
existing core input decoder/host, UI application/event loop/menu behavior, web xterm.js adapter,
tests, consumer documentation, and plugin-impact governance. The modification set is the new
feature's plan, roadmap, traceability graph, and preflight report.

**AR-2:** The public policy name is `FunctionKeyFallback`, with values `'number-row'` and `'none'`.
The application shell selects `'number-row'` when omitted; lower-level decoder APIs never rewrite a
valid Alt chord.

**AR-3:** The existing `ESC_TIMEOUT_MS = 50` remains the byte-stream disambiguation boundary. No
second, longer-lived Escape-prefix state machine is introduced.

**AR-4:** Modifier preservation applies when the terminal protocol represents modifiers
unambiguously. A fallback alias produces an unmodified canonical function key because the Alt bit is
the fallback introducer, not an application modifier.

**AR-5:** Browser interception is limited to the focused terminal, handles `keydown` once, prevents
the browser default where possible, and returns control in the form required to prevent duplicate
xterm.js data.

**AR-6:** Documentation must state that OS, firmware, terminal, and browser reservations can prevent
a physical key from reaching JSVision; the number-row fallback is the portable accessibility
contract.

**AR-7:** The compatibility impact is intentional and reversible per application. Existing
applications that require genuine Alt+number-row bindings configure `'none'`.

**AR-8:** The allowlist is grounded in published xterm/terminfo-compatible legacy forms, Linux
virtual-console forms, and Kitty's specified CSI-u functional-key identifiers. Unknown or malformed
sequences retain the decoder's bounded drop/resynchronization behavior.

**AR-9:** F13 and later, keypad function aliases, event repeat/release reporting, protocol
negotiation, and wholesale enhanced-keyboard activation remain outside this plan.

**AR-10:** `yarn verify` is the authoritative final gate from `AGENTS.md`. Any path reported by
`tools/jsvision-plugin-impact.json` triggers reference review and `yarn plugin:update`; every SDK
change also runs `yarn plugin:check`.
