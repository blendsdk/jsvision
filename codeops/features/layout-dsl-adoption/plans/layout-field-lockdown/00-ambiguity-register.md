# Ambiguity Register: layout-field-lockdown

Every decision below was put to the maintainer with grounded options and decided by them. Rows
marked *(measured)* were settled by evidence gathered in a reverted spike — the field was flipped
to `readonly`, `ui` was rebuilt so downstream packages saw the new `.d.ts`, and `tsc --noEmit` ran
per package. The tree was restored and `dist` rebuilt afterwards.

## Register

| AR-1 | Design *(measured)* | How deep the lockdown goes. TypeScript's `readonly` is shallow, so `readonly layout: LayoutProps` and `readonly layout: Readonly<LayoutProps>` differ in what they forbid | (a) deep — `Readonly<LayoutProps>`; (b) shallow — `readonly LayoutProps` | **(a) deep** | Measured: both variants produce **exactly 21 errors** in `ui` — adding `Readonly<>` caught *nothing* extra until the subclass escape hatches (AR-3) were closed, after which it rose to 31 and revealed every in-place `layout.rect =` mutation. Shallow leaves all **32** mutation sites legal, so the field would be only half locked: the object it hands back stays freely mutable and the invariant is advisory. (b) was the smaller plan (~81 sites) but does not deliver the issue's stated goal |
| AR-2 | Design *(measured)* | `setLayout` (`packages/ui/src/view/view.ts:254`) assigns the very field being frozen — `this.layout = { ...this.layout, ...patch }` — so it must change | (a) mutate in place via `Object.assign`; (b) private backing field + public getter; (c) keep replace, write through a cast | **(a) `Object.assign(this.layout, patch)`** | Keeps `layout` a real field, so all **10** subclass `override layout = {…}` initializers keep working with only an `override readonly` edit. (b) is what the issue's own body implies and is the trap: **a class field cannot override an accessor**, so it breaks all ten at compile time and forces redesigning those classes — this is precisely why P1 shipped without a getter. (c) preserves today's semantics exactly but puts a banned cast in the one place that must be trustworthy. **Identity risk checked and cleared**: the only holder of the layout object is `reflow.ts:78` (`props: view.layout`), and that `LayoutBox` is rebuilt on every pass; `stack.ts:139` reads `.rect` only to compare values. No long-lived reference exists, so in-place mutation is unobservable today |
| AR-3 | Design *(measured)* | The 10 `override layout: LayoutProps = {…}` subclass redeclarations | (implicit in AR-1) | **All 10 become `override readonly layout: Readonly<LayoutProps>`** | These are **silent escape hatches**, not merely affected sites: a subclass that redeclares the field without `readonly` re-opens it, and the compiler says nothing. Measured — `Window` (`window/window.ts:81`) does exactly this, which is why `win.layout.rect = …` in `window.ts`, `desktop/gestures.ts` and `desktop/arrange.ts` compiled clean through **both** naive flips. Ten further sites appeared only once the hatches were shut. Any lockdown that leaves them open reports a false all-clear |

(b) creates a second blessed writer, reintroducing the "which one do I use" ambiguity the issue exists to remove. (c) is a genuine API design exercise that fits ~10 window sites and poorly fits the other 22 |

## Latent-defect policy (from AR-5's neighbourhood, decided with it)

Three of #132's errors sit in test files that **currently pass** — 4× `TS2722` *"cannot invoke an
object which is possibly undefined"* in `test/datagrid-showcase.spy-source.spec.test.ts`, a
`TS2554` arity error in `test/recipes.smoke.spec.test.ts`, a `TS2345` wheel-event shape mismatch in
`test/probe-readout.impl.test.ts`, and 2× `TS2740` stream-type mismatches in
`test/probe-nontty.spec.test.ts`.

**Decision: investigate and fix each properly; do not silence them.** A type error in a passing
test means either the fixture is wrong or the assertion is weaker than it reads — and these are
spec oracles, where a test that passes by accident is worse than no test. Each gets a one-line
verdict recorded during execution. Minimal non-null assertions were rejected: they convert a
signal into silence in exactly the files meant to be the oracle.
