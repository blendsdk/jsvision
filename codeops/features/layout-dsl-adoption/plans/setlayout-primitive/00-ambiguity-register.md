# Ambiguity Register — setlayout-primitive

> **Gate**: ✅ GATE PASSED — every item resolved with an explicit user decision, zero deferred.
> **Feature**: layout-dsl-adoption · **Implements**: GH #117

Decisions taken before any plan document was written. Each records what was ambiguous, the options
that were genuinely viable, and the ruling.

| # | Item | Decision | Status |
|---|------|----------|--------|
| AR-1 | How far through the issue's P1→P4 phasing does this plan go? | **P1–P3; P4 deferred.** P4 (read-only `layout`) needs *all* 138 writes gone, which needs #110's remainder, #112 and #114 finished — and a solution for the 11 field initializers (AR-3). None of those is in hand, so P4 is genuinely blocked rather than merely large. It becomes its own tracked follow-on. | ✅ Resolved |
| AR-2 | Which write sites migrate to `setLayout` here? | **The DSL internals (13 sites) + the 7 `this.layout =` self-layout sites.** Exactly the issue's P2 wording. The remaining ~118 composition writes belong to the open adoption issues that already own them; converting them here would duplicate that work, balloon the diff across 7 packages, and touch sites the epic is about to delete outright. | ✅ Resolved |
| AR-3 | The 11 subclasses that set layout via a **field initializer** (`override layout: LayoutProps = {…}`). | **Left declarative.** They cannot exhibit the footgun — a freshly-constructed object has no sibling props to drop — and converting them to constructor calls costs readability, forces constructors onto classes that have none, and shifts initialization order for no behavioural gain. Recorded precisely as the blocker P4 must solve. | ✅ Resolved |
| AR-4 | Verify command at every phase gate. | `TUI_SKIP_PERF=1 yarn verify && yarn workspace @jsvision/examples test:e2e`. The second half is load-bearing: `yarn verify` runs only the `unit` project, and `ui` changes reach the demos through the built `dist`. | ✅ Resolved |
| AR-5 | `setLayout` auto-invalidates, but the DSL taggers do **not** today — so converting them changes runtime behaviour on already-mounted views. | **In scope, treated as a latent bug fix.** At build time `host` is `null`, so the overwhelming majority of calls are unchanged no-ops. It only bites where a tagger re-runs on a mounted view (`filter-popup.ts:285`), and there the current silence *is* the defect: the view re-tags and never reflows. Pinned by ST-S5. | ✅ Resolved |
| AR-6 | Mark the `layout` field `@deprecated` in P1? | **No — prose only.** JSDoc states `setLayout` is the preferred write path without the tag. A `@deprecated` tag renders ~118 remaining call sites struck through in every editor and in the generated TypeDoc, which is a lot of noise for a removal (P4) that is explicitly deferred and not yet schedulable. | ✅ Resolved |
| AR-7 (derived) | Does P1 introduce the `get layout()` / `_layout` pair the issue's design sketch shows? | **No — `layout` stays a plain public field.** Forced by AR-3: a TypeScript class field cannot override a base-class accessor, so introducing the getter breaks all 11 field initializers *immediately* rather than at P4. `setLayout` merges into the public field. The getter is part of P4's problem statement, not P1's. | ✅ Resolved |
| AR-8 (derived) | Where does the P3 replace-semantics audit's output live? | **In this plan's `03-02` document**, not in shipped source. Per CLAUDE.md the audit is a working artifact; a per-site finding that matters to a *reader of the code* becomes a plain-language comment, and everything else stays here. | ✅ Resolved |
| AR-9 (derived) | `packages/spike-data-studio` holds 13 `.layout =` writes. | **Excluded.** CLAUDE.md marks it an inert throwaway spike with no build/typecheck/test wiring, slated for deletion after the Data Studio decision. Migrating it would be work on code that is scheduled to disappear. Recorded as a P4 precondition: it must be deleted or migrated before `layout` can go read-only. | ✅ Resolved |
